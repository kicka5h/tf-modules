# az-ipam

Algorithmically generates non-overlapping IP address allocations for Azure environments, VNets, and subnets from a root CIDR block, while avoiding conflicts with existing infrastructure.

## The Problem

- Teams manually calculate CIDRs and make mistakes
- VNets overlap across environments
- Nobody knows what's already allocated in the tenant
- Adding a new VNet means auditing every existing address space

## How It Works

1. **Discover** existing allocations using `scripts/discover-ip-allocations.sh` (queries Azure Resource Graph)
2. **Define** your allocation plan using `cidrsubnet()` math (newbits + index)
3. **Validate** that new allocations don't overlap with existing ones
4. **Output** CIDRs that feed directly into the `az-virtual-network` module

All CIDRs are derived from a single root block using `cidrsubnet()`, making overlaps mathematically impossible within new allocations. The `reserved_cidrs` input prevents conflicts with pre-existing infrastructure.

## Usage

### Step 1: Discover existing allocations

```bash
# Requires: az login
./scripts/discover-ip-allocations.sh > reserved_cidrs.json

# Or generate tfvars directly
./scripts/discover-ip-allocations.sh --tfvars > reserved.auto.tfvars
```

The script queries Azure Resource Graph for all VNets, subnets, and public IPs across every subscription in the tenant.

### Step 2: Define your allocation plan

```hcl
module "ipam" {
  source     = "../az-ipam"
  root_cidrs = ["10.0.0.0/8"]

  # From discovery script
  reserved_cidrs = [
    "10.100.0.0/16",  # legacy datacenter
    "172.16.0.0/12",  # on-premises
  ]

  allocations = {
    dev = {
      cidr_newbits = 8    # /16 from /8
      cidr_index   = 0    # 10.0.0.0/16
      vnets = {
        hub = {
          cidr_newbits = 4  # /20 from /16
          cidr_index   = 0  # 10.0.0.0/20
          subnets = {
            GatewaySubnet       = { cidr_newbits = 4, cidr_index = 0 }  # /24
            AzureFirewallSubnet = { cidr_newbits = 4, cidr_index = 1 }  # /24
          }
        }
      }
    }
    prod = {
      cidr_newbits = 8
      cidr_index   = 3    # 10.3.0.0/16
      vnets = {
        hub = {
          cidr_newbits = 4
          cidr_index   = 0
          subnets = {
            GatewaySubnet       = { cidr_newbits = 4, cidr_index = 0 }
            AzureFirewallSubnet = { cidr_newbits = 4, cidr_index = 1 }
          }
        }
      }
    }
  }
}
```

### Step 3: Feed into az-virtual-network

```hcl
module "vnets" {
  source              = "../az-virtual-network"
  resource_group_name = var.resource_group_name
  location            = var.location

  vnets = {
    for key, vnet in module.ipam.vnets_for_module : key => {
      name          = "vnet-${key}"
      address_space = vnet.address_space
      subnets = {
        for sk, sv in vnet.subnets : sk => {
          address_prefixes = sv.address_prefixes
        }
      }
    }
  }
}
```

## Configuration Reference

### How cidrsubnet math works

`cidrsubnet(prefix, newbits, index)` divides a CIDR into smaller blocks:

| Root | newbits | Result size | Number of blocks | Example |
| --- | --- | --- | --- | --- |
| `/8` | 8 | `/16` | 256 | One /16 per environment |
| `/16` | 4 | `/20` | 16 | One /20 per VNet |
| `/20` | 4 | `/24` | 16 | One /24 per subnet |
| `/20` | 2 | `/22` | 4 | Larger subnets (AKS, app tiers) |

The `cidr_index` selects which block within the division (0-indexed).

### Variables

| Variable | Type | Description |
| --- | --- | --- |
| `root_cidrs` | `list(string)` | Top-level CIDR blocks (e.g., `["10.0.0.0/8"]`) |
| `reserved_cidrs` | `list(string)` | Existing CIDRs to avoid. Generated by the discovery script. |
| `allocations` | `map(object)` | Hierarchical allocation plan: environment -> VNet -> subnet |

### Allocation object

| Field | Type | Description |
| --- | --- | --- |
| `cidr_newbits` | `number` | Bits to add to the parent CIDR |
| `cidr_index` | `number` | Index within the parent (0-based) |
| `vnets` | `map(object)` | VNets within this environment (same fields + `subnets`) |
| `subnets` | `map(object)` | Subnets within this VNet (same fields, no children) |

### Outputs

| Output | Description |
| --- | --- |
| `environment_cidrs` | Map of environment name to CIDR |
| `vnet_cidrs` | Map of `env-vnet` key to CIDR and address_space |
| `subnet_cidrs` | Map of `env-vnet-subnet` key to CIDR and address_prefixes |
| `vnets_for_module` | Pre-built map for direct use with `az-virtual-network` |
| `has_overlaps` | Boolean — true if any new allocation overlaps with reserved CIDRs |
| `overlapping_cidrs` | List of conflicting CIDR pairs |
| `summary` | Counts of environments, VNets, subnets, and overlaps |

## Discovery Script

`scripts/discover-ip-allocations.sh` queries Azure Resource Graph for:

| Resource | What it collects |
| --- | --- |
| VNets | Name, resource group, subscription, location, address prefixes |
| Subnets | VNet name, subnet name, address prefix |
| Public IPs | Name, IP address, allocation method, SKU |

### Formats

| Flag | Output | Usage |
| --- | --- | --- |
| (none) | JSON | `./scripts/discover-ip-allocations.sh > discovery.json` |
| `--tfvars` | Terraform tfvars | `./scripts/discover-ip-allocations.sh --tfvars > reserved.auto.tfvars` |

The `--tfvars` format outputs `reserved_cidrs` and `existing_vnets` variables that can be loaded directly by Terraform.

### Prerequisites

- Azure CLI installed and logged in (`az login`)
- Reader access across target subscriptions
- `az graph` extension (auto-installed by Azure CLI)

## Overlap Detection

The module checks every new allocation against every reserved CIDR:

- If `has_overlaps = true`, the `overlapping_cidrs` output shows which pairs conflict
- Use this in CI to fail the plan:

```hcl
resource "null_resource" "overlap_check" {
  count = module.ipam.has_overlaps ? "OVERLAP_DETECTED" : 0
}
```

Or check in the pipeline:

```bash
terraform plan -out=tfplan
OVERLAPS=$(terraform output -json has_overlaps)
if [ "$OVERLAPS" = "true" ]; then
  echo "ERROR: IP address overlap detected"
  terraform output -json overlapping_cidrs
  exit 1
fi
```
